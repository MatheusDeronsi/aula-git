Primeiramente Fora Temer, vamos falar sobre Controle de Versão

1. Controle de versão

O que é esse tal Controle de Versão?

O controle de versão é um sistema que registra as mudanças feitas em um arquivo ou um conjunto de arquivos ao longo do tempo de forma que você possa recuperar versões específicas.

Tá mas e o exemplo visual...

2. E esse GIT, que tem ele nessa história?

GIT é um sistema de controle de versão de arquivos. Através dele podemos desenvolver projetos na qual diversas pessoas podem contribuir simultaneamente no mesmo, editando e criando novos arquivos e permitindo que os mesmos possam existir sem o risco de suas alterações serem sobrescritas.

Em outras palavras...

O GIT é a ferramenta que vai te auxiliar a criar um ambiente colaborativo sem cagar causar danos no código de outro desenvolvedor ou criar 420 versões de um mesmo código - no DropBox daquele seu trabalho de Algoritmos em grupo - por exemplo.

Só uma trivia...

O GIT foi projetado e desenvolvido pelo mesmo criador do Linux, o sr. Linus Torvalds.

3.Vamos começar então um projeto com controle de versão?

Vamos a instalação...

No terminal do Linux (Debian/Ubuntu) digite:

$ sudo apt-get install git

No Windows acesse estes links em seu navegador e baixe:

Versão 32 bits -> http://bit.ly/2k5wbw5

Versão 64 bits -> http://bit.ly/2ku0v79

Agora vamos começar com os comandos básicos...

Crie uma pasta em qualquer lugar de sua máquina para iniciar seu projeto:


Meu-Projeto

No : abra o terminal clicando no símbolo do Ubuntu e digitando 'terminal'


No : clique no botão 'Iniciar' e procure pelo programa GIT na barra de programas do Windows, execute-o, um terminal será aberto também.

Pelo terminal de ambos, navegue até sua pasta:

$ cd ../caminho-do-projeto/Meu-Projeto

Certifique-se que chegou a pasta corretamente.

Pelo terminal, já dentro da pasta, digite:

$ git init

Uma mensagem de retorno deve mostrar o sucesso na criação do seu repositório GIT:

Initialized empty Git repository in /caminho-do-projeto/Meu-Projeto/.git/

Pronto! Você acaba de criar um repositório GIT e
já pode começar a versionar seu código!
Vamos agora adicionar um arquivo ao nosso repositório pra entender o versionamento na prática.

Crie um arquivo qualquer dentro da sua pasta do projeto, pode ser um .txt mesmo:

/
Meu-Projeto / meu-arquivo.txt

Abra o arquivo de texto em um editor qualquer e escreva uma mensagem dentro, pode ser o famigerado: 'Olá iMundo! Salve o arquivo .txt.

Com o arquivo alterado e salvo, volte ao terminal e, de dentro de seu projeto, digite:

$ git status

Uma mensagem de retorno deve mostrar que o novo arquivo foi detectado.

Arquivos não monitorados:
new file: meu-arquivo.txt

Agora seu repositório sabe que existe um arquivo novo, que torna o repositório diferente do estado anterior. Vamos adicionar esse novo arquivo oficialmente ao nosso repositório.

Volte ao terminal e, de dentro de seu projeto, digite:

$ git add meu-arquivo.txt

Uma mensagem de retorno deve mostrar que seu arquivo foi adicionado a pilha de arquivos a serem enviados ao respoitório.

Mudanças a serem submetidas:
new file: meu-arquivo.txt

Volte ao terminal e digite:

$ git commit -m "Meu primeiro commit!"

Uma mensagem de retorno deve mostrar que seu arquivo foi enviado ao seu repositório com sucesso!

[master 1d922c1] Meu primeiro commit com o arquivo novo. 1 file changed, 0 insertions(+), 0 deletions(-) create mode 100644 meu-arquivo.txt

Pronto! Seu arquivo foi enviado com sucesso para o repositório da sua máquina, ou seja, seu computador já sabe que este novo arquivo também faz parte da 'árvore de arquivos' do projeto.
Vamos gerar um pequeno conflito de versões então?

Abra o arquivo de texto criado na pasta Meu-Projeto e altere a mensagem dentro dele, vamos colocar algo do tipo: 'Olá iMundo! Segundas-feiras são tão ~legais~! Salve o arquivo .txt novamente.

Volte ao terminal e digite:

$ git status

Uma mensagem de retorno deve mostrar que seu arquivo foi modificado da versão atual que se encontra em seu repositório.

Mudanças not staged para commit:
modified: meu-arquivo.txt

Vamos então commitar a versão 2 do mesmo arquivo...

Volte ao terminal novamente e digite para verificar as alterações:

$ git status

Agora adicione o arquivo a pilha de envio, dessa ver usaremos o comando um pouco diferente:

$ git add .

Uma mensagem de retorno deve mostrar que seu arquivo foi adicionado a pilha de arquivos a serem enviados ao respoitório.

Mudanças a serem submetidas:
modified: meu-arquivo.txt

Vamos agora commitar a versão 2:

$ git commit -m "Meu segundo commit com as alterações novas."

Uma mensagem de retorno deve mostrar que seu arquivo modificado foi enviado com sucesso.

[master bd35149] Meu segundo commit com as novas alterações. 1 file changed, 2 insertions(+)

6.Git Hooks - wtf?

Git hooks são scripts que rodam automaticamente cada vez que você realiza alguma ação em particular no repositório Git.

Casos comuns do uso de Git Hooks são encorajar alguma política de commit, alterar algo do ambiente dependendo do estado do repositório ou implementar algum workflow de Integração Contínua.

Como funciona?

Todo repositório Git ao ser criado, gera um pasta .git, que, em geral, é oculta pelos sistemas operacionais. Dentro dessa pasta que é feita toda a mágica de versionamento, logs, separação de branches e etc.

Vamos modificar a menasgem de commit atual apenas para ilustrar o funcionamento.

$ ls -la .git/hooks

Renomeie o arquivo .git/hooks/commit-msg.sample removendo o '.sample'

$ mv .git/hooks/commit-msg.sample ./git/hooks/commit-msg

Agora vamos adicionar uma mensagem custom para o commit

$ echo " E esse commit aí?? " >> .git/hooks/commit-msg
